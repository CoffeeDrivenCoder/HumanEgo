# G1 末端位姿控制与 HumanEgo 适配汇报简版

## 目标

HumanEgo 原始模型是在 ego/Aria 数据上训练的，推理输出的是相机坐标系下的手部末端目标位姿和夹爪开合。我们要做的是把这套输出转换成 G1 机器人能够执行的末端位姿控制命令。

核心问题不是单纯跑模型，而是打通这一条链路：

```text
G1 RGB-D 输入
-> HumanEgo 推理输出 T_hand_in_cam
-> 转成 G1 TCP / link7 目标
-> 调用 G1 set_end_effector_pose_control
-> 控制真实末端运动
```

## 适配分步

### 1. 获取 G1 视觉输入

我们先确认了 G1 头部相机可以稳定提供 HumanEgo 需要的 RGB-D 输入：

- RGB：`head`
- Depth：`head_depth`
- 深度单位：`uint16`，按 mm 转 m
- 图像尺寸：`800 x 1280`

同时实现了 `G1Camera`，用于把 G1 的相机数据包装成 HumanEgo 推理需要的输入格式。

### 2. 获取相机内外参

G1 的相机参数通过机器人端参数服务读取：

```text
http://10.42.0.101:8849/camera_parameters
```

已确认可以拿到：

- 相机内参 `K`
- 固定外参 `T_head_pitch_camera`

这里的关键点是：参数服务返回的 `T` 不是最终的 `T_base_camera`，而是头部 pitch link 到相机的固定外参。

### 3. 计算相机到机器人 base 的变换

因为头和腰会影响相机在 base 下的位置，所以最终外参需要这样计算：

```text
T_base_camera = T_base_head_pitch(q) @ T_head_pitch_camera
```

其中：

- `T_head_pitch_camera` 来自参数服务
- `T_base_head_pitch(q)` 通过 `corobot` 的头腰 FK 得到

我们已经用真机数据验证了这个方向是正确的：把右手 TCP 投影回头部 RGB 图像，投影点和实际夹爪位置一致。

### 4. 定义 G1 末端 TCP

G1 SDK 的 `motion_status` 里读到的是：

```text
arm_right_link7
```

但 HumanEgo 控制的更接近夹爪中心/TCP，所以需要从 URDF 中补上固定变换：

```text
link7 -> gripper center / TCP
```

当前使用的右手 TCP 定义：

```text
T_tcp_in_link7:
  位置：沿 link7/夹爪方向偏移 0.14308m
  旋转：与 G1 Omnipicker URDF 中 gripper center 对齐
```

这样就可以从 G1 当前状态得到：

```text
T_link7_in_base
-> T_tcp_in_base
-> T_tcp_in_cam
```

### 5. 对齐 HumanEgo 手部 frame 和 G1 TCP frame

HumanEgo 输出的是 Aria/ego 数据里的手部 frame，不一定等同于 G1 的 TCP frame，因此需要一个对齐矩阵：

```text
T_align = T_hand_in_tcp
```

第一版已经写入配置，用于把 G1 TCP frame 映射到 HumanEgo 训练时的右手 frame。后续如果真机动作方向有系统性偏差，再调整这个矩阵。

### 6. 推理输出转 G1 控制目标

HumanEgo 输出：

```text
T_hand_target_in_cam
```

转换成 G1 控制目标的流程是：

```text
T_hand_target_in_cam
-> 乘 T_align 的逆，得到 T_tcp_target_in_cam
-> 通过 T_base_camera 转到 base 坐标
-> 扣除 link7 -> TCP 固定变换
-> 得到 T_link7_target_in_base
-> 调用 set_end_effector_pose_control
```

夹爪部分也已经确认：

```text
G1 move_gripper: 0~120
HumanEgo gripper: 0~1
0 = 完全张开
1 = 完全闭合
```

所以夹爪可以直接用：

```text
g1_value = humanego_value * 120
```

## 已完成验证

目前已经完成：

- G1 RGB-D 数据读取。
- G1 相机内参读取。
- G1 头部相机固定外参读取。
- `corobot` 头腰 FK 接入。
- `T_base_camera = T_base_head_pitch @ T_head_pitch_camera` 真机投影验证。
- `arm_right_link7 -> gripper center/TCP` 的 URDF 变换整理。
- 当前 TCP 在相机坐标系下的计算。
- HumanEgo hand frame 与 G1 TCP frame 的第一版 `T_align`。
- 只读版 `G1RobotArm` adapter。
- `set_end_effector_pose_control` 的 frame 语义已完成真机验证。

## 末端控制验证结果

真机上已经验证：

```text
set_end_effector_pose_control
```

接收的是：

```text
base_link 坐标系下的 arm_right_link7 pose
```

调用格式为：

```text
control_group = ["right_arm"]
right_pose = {x, y, z, qx, qy, qz, qw}
```

验证数据：

```text
发送目标位移：base_link +Z 0.0100m
读回实际位移：[+0.00036, -0.00004, +0.00931]m
```

实际运动主方向和目标一致，误差在毫米以内，因此可以确认 HumanEgo 输出经过坐标转换后，可以送入 G1 末端位姿控制接口。

## 当前结论

G1 与 HumanEgo 的主要适配问题已经被拆成了明确的坐标链：

```text
HumanEgo camera-frame hand target
-> G1 camera-frame TCP target
-> G1 base-frame link7 target
-> G1 SDK end-effector pose control
```

相机、外参、FK、TCP 定义、夹爪开合和 G1 末端控制 frame 语义都已经完成第一轮真机验证。下一步可以进入 HumanEgo 推理 dry-run：先打印模型输出和 G1 link7 目标，不直接连续控制；数值稳定后再接入小步长闭环控制。
